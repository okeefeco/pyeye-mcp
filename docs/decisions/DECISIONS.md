## 2026-06-28 — trace() surfaces interior callee-resolution loss (unresolved_call_sites) (#488)

**Friction:** `unresolved_roots` made the ROOT layer honest, but the same goto non-determinism (#419) plus inherently-dynamic calls leave INTERIOR callee walks incomplete — and `trace` silently dropped the per-node unresolved-call-site count that `expand(callees)` already reports. A populated, `unresolved_roots: []` subgraph could still be quietly lossy and read as complete.
**Decision:** Carry `EdgeResult.unresolved_call_sites` through `_single_hop` and surface a `{source_handle: count}` map on the Subgraph. Chose PRESENT-IFF-CALLEES-TRACED (not always-present like `unresolved_roots`): call sites are only measured when `callees` is followed, so an absent key is the honest "not measured" (absence-vs-zero), `{}` = traced and all resolved, and only count > 0 nodes appear. Same invariant as `unresolved_roots` applied correctly to a *conditionally*-measured fact rather than an always-measured one. Does not cover non-callees edges (mostly deterministic AST) or whole-walk determinism (#419 / #333) — scoped explicitly in the docstrings.
**Anchor:** `pyeye.mcp.operations.trace.trace`, `pyeye.mcp.operations.trace._single_hop`; response_linter rule T.7; #488 (PR pending).
**Verify:** gold — `trace("mypackage._core.callees_fixture.orchestrate", ["callees"], analyzer)["unresolved_call_sites"] == {"mypackage._core.callees_fixture.orchestrate": 1}`; a members-only trace omits the key; linter T.7 rejects non-dict / non-int / count < 1. Covered by `tests/unit/mcp/operations/test_trace.py::TestTraceUnresolvedCallSites` and `tests/conformance/test_linter_adversarial_trace.py::TestTraceUnresolvedCallSites`.

## 2026-06-28 — trace() surfaces unresolved roots as an always-present honesty field (#488)

**Friction:** trace()'s first (cold) call returned a clean `nodes: {}` for a root that silently failed to resolve (#488 / #419 cold-start non-determinism) — an agent reads that as "this root has no neighbours," a false-confident negative in the primitive whose entire value is honesty.
**Decision:** Add `unresolved_roots` to the Subgraph. Chose ALWAYS-PRESENT (`[]` when every root resolved, the failed handles otherwise) over the initially-implemented conditional / non-empty-only form: root resolution is always measured, so under the absence-vs-zero invariant absence must mean "not measured," never "all resolved." Made it symmetric with its sibling `unsupported_edges` (same Subgraph, same input-honesty role), NOT with the conditional `report_issues`. Absence is therefore a conformance violation (a broken trace), not a signal. Rejected: conditional presence — it overloads absence with the everyday "all resolved" value and makes a field-dropping bug indistinguishable from a healthy trace.
**Anchor:** `pyeye.mcp.operations.trace.trace`; response_linter rule T.6; #488 (PR pending).
**Verify:** gold — `trace("does.not.exist.Sym", ["members"], analyzer)["unresolved_roots"] == ["does.not.exist.Sym"]`; a resolvable root yields `["unresolved_roots"] == []`; linter T.6 raises on a trace response missing the key. Covered by `tests/unit/mcp/operations/test_trace.py::TestTraceUnresolvedRoots` and `tests/conformance/test_linter_adversarial_trace.py::TestTraceUnresolvedRoots`.

## 2026-06-28 — Reverse references are a sanctioned LSP handoff, not a dead-end deferral (#489)

**Friction:** The python-explore skill's headline honest-limits rule told the agent reverse references "cannot be answered reliably yet" (deferred to #333) and offered only forward edges — a dead end. But the harness almost always already carries a Pyright/Pylance-backed LSP tool that answers exactly "who calls / references this," and pyeye's `resolve`/`inspect`/`resolve_at` already emit the def-site `location` (`file:line:column`) an LSP reference query needs. Treating the deferral as a hard wall left a real, composable capability unused — and an agent with no sanctioned path is the one most tempted to cheat with grep or the legacy reference tools.
**Decision:** Reframe pyeye's reverse-reference deferral as a sanctioned *handoff*: pyeye drives orientation/structure/forward facts; delegate the inbound questions to the editor's LSP via `resolve` → def-site `location` → LSP `findReferences` / `incomingCalls` / `goToImplementation` → `resolve_at` to map results back into handle space. Delegation is explicitly *not* faking; the grep + legacy-tool (`find_references`/`get_call_hierarchy`) ban and the absence-vs-zero (`edge_counts` omits, never `0`) rule stay intact. Added a "roles, not redundancy" rule: never consult pyeye and the LSP for the *same* fact (Jedi vs Pyright infer independently and can disagree — two oracles voting erodes the trust a single honest answer is for). When no LSP is in reach, fall back to the prior honest refusal + forward edges (`callees`/`imported_by`/`imports`). Rejected: keeping the flat "not available yet" wall (leaves the capability unused and invites grep cheats).
**Anchor:** `skills/python-explore/SKILL.md` ("Honest Limits — pyeye Has No Reverse References (Delegate to Your LSP)") ; #489 (refs #333).
**Verify:** PARTIAL — gold: `tests/test_python_explore_skill_conformance.py` (6/6) guards the skill's supported-edge list against the live edge registry, so the reverse edges stay documented-as-deferred and the absence-vs-zero invariant holds. The behavioural claim — that an agent actually delegates to the LSP rather than grepping — is human-observed [unverifiable mechanically].

## 2026-06-27 — `_search_all_scopes` resolves via the AST name-index, not `jedi.Project.search` (#457)

**Friction:** `jedi.Project.search(name, all_scopes=True)` silently truncates at Jedi's internal 30-parsed-file cap (`_PARSED_FILE_LIMIT`), so high-frequency names lose real definitions: on django, `resolve("Field")` returned a single false-confident hit instead of the project's three `Field` classes (#457). Raising the cap only moves the cliff — the project-wide reverse-search shape is the same disease #405 already cured for `find_subclasses`.
**Decision:** `_search_all_scopes` now looks names up in a whole-project, non-evicting, invalidate-and-rebuild AST name→definitions index (`project_graph`), built once via `extract_definitions` and filtered to the requested scope **at lookup, never at build** (a scope-keyed build would poison the project-keyed cache). The index holds modules (incl. `__init__.py` = regular packages), top-level classes/functions/statements, and nested classes; methods/attributes are reached via the parent (`_enclosed_definitions`, pure AST), never indexed by bare name. Index module names carry the root-package prefix when the project root is a package (`_index_module_name`, matching Jedi's `full_name`), while `_get_import_path_for_file` stays prefix-free for `find_reexports`. Consequence: `resolve._resolve_bare_name` no longer early-returns on empty `find_symbol` matches, so PEP 420 namespace-package anchoring (#444) still fires when the index yields nothing (a namespace package has no file to index). Determinism over Jedi reverse search (#449). **Scope:** `_search_all_scopes` only — `inspect`/`canonicalization`/`search_all_projects` still call `project.search` for other paths (separate, out of scope).
**Anchor:** `JediAnalyzer._search_all_scopes` ; `pyeye.analyzers.project_graph.get_name_index` ; `resolve._resolve_bare_name` ; commits `4341b47`/`fb9cbe4` ; #457 (closes #450, seeds #397)
**Verify:** gold — `tests/test_search_all_scopes_index.py::test_returns_every_definition_past_the_30_file_cap` (40 modules → 40 defs, was 30 on `main`); `tests/test_resolve_completeness_457.py` (`resolve` ambiguous with all N, not truncated); django dogfood: `resolve("Field")` → the 3 canonical `Field`s + a nested one, `resolve("django.forms.fields.Field")` → real line 84.

## 2026-06-23 — uv lockfile is authoritative everywhere; pinned per context (#481, follows #479)

**Friction:** Behind a PyPI mirror, bare `uv run` in pre-commit re-resolved and rewrote `uv.lock`, tripping pre-commit's framework-level "files were modified by this hook" check and failing `git checkout`/`commit` (#479). The deeper issue: nothing guaranteed every environment built from the *exact* locked config — bare `uv run`/`uv sync` may silently re-resolve (a mirror rewrite, or CI drifting from a stale lock), so "use the exact config" could quietly degrade.
**Decision:** Make the lockfile authoritative everywhere, pinned per context by how strict the no-deviation must be. **Hooks** use `uv run --frozen` (use the lock as-is; never block, never rewrite — fresh-clone friendly) (#479). **CI** pins `uv sync --locked` on its setup steps (assert `uv.lock == pyproject.toml`, fail on drift); subsequent `uv run` runs against the just-verified env. Resolution happens only via a deliberate `uv lock`/`uv add`. The `--frozen`-for-hooks vs `--locked`-for-CI split is intentional: hooks must never block local work, while the CI gate must fail loudly on drift (`--frozen` in CI would silently run a stale lock and hide it). A workflow-level `UV_LOCKED=1` env var was tried first (one declaration, can't-erode) but **reverted**: `UV_LOCKED` is mutually exclusive with `--frozen`, and CI's `pre-commit` job runs the `--frozen` hooks, so uv aborted with `UV_LOCKED cannot be used with --frozen` (PR #482 CI; an empty per-job override does NOT clear it — uv treats the var as present). A `--locked` *flag* on the standalone `uv sync` never shares an environment with the `--frozen` hook runs, so they can't collide. Rejected: `--frozen` in CI (hides drift); workflow-level `UV_LOCKED` (collides with the hooks).
**Anchor:** `uv sync --locked` in `.github/workflows/{ci,release,build-deploy-docs}.yml`; `uv run --frozen` entries in `.pre-commit-config.yaml`; guard tests `tests/test_ci_uv_pinned.py::test_ci_uv_sync_is_locked` and `tests/test_precommit_hook_config.py::test_all_uv_run_entries_are_frozen`; CONTRIBUTING.md "Dependency lockfile policy"; #481, #479.
**Verify:** gold — `test_ci_uv_sync_is_locked` asserts every `uv sync` in a workflow carries `--locked`; `test_all_uv_run_entries_are_frozen` asserts no pre-commit hook entry uses a bare `uv run`. Behavioral (proven locally): with a dependency added to `pyproject.toml` but `uv.lock` left unchanged, `uv sync --locked` fails with "The lockfile at `uv.lock` needs to be updated, but `--locked` was provided"; `uv run --frozen python scripts/refresh_version.py` on a fresh clone leaves `uv.lock` unchanged; and `UV_LOCKED=1 uv run --frozen …` aborts (the collision that forced the flag-over-env choice).

## 2026-06-23 — outline `max_nodes` truncation: reserve-before-expand + `member_count` signpost (#358)

**Friction:** When the node budget was exhausted *mid-enumeration* of a container's children, `outline` `break`-ed the loop and assigned the **partial** `children` list with no truncation marker — a consumer read a partial set as the *complete* direct-member set (a Contract-1 honesty violation). The conformance linter could not catch it (a partial list is shape-valid), and the worst case — all dropped siblings are leaves — left ZERO truncation markers anywhere in the tree (#358).
**Decision:** Reserve-before-expand: a container is expanded only if the remaining budget can admit ALL its direct members; otherwise it is cut off **whole** (`truncated: "max_nodes"`, `children` omitted) and carries a fresh `member_count` (the count of withheld direct members). Never a partial `children` list. `member_count` is added to the `max_nodes`-family truncations only; `external`/`max_depth` caps are left as-is (no peek, no count). Rejected option-1 (expand-then-discard-partial): same output but wastes stub builds and would enqueue orphan nodes. Consequence accepted: a container larger than the remaining budget now surfaces zero members + a count, recoverable by raising `max_nodes` or targeting the subtree.
**Anchor:** `pyeye.mcp.operations.outline.outline` (reserve check + member_count); linter rule O.5 in `tests.conformance.response_linter._check_outline_tree_node`; #358; spec `docs/superpowers/specs/2026-06-22-bounded-response-firehose-contract-design.md` §4.
**Verify:** gold — `outline("pkg.mod", JediAnalyzer("tests/fixtures/outline_flat_leaves"), max_nodes=3)` returns the root with `truncated == True`, `truncation_reason == "max_nodes"`, `member_count == 5`, and NO `children`; for every budget over `tests/fixtures/nested_class_inspect`, no node carrying `children` is missing a direct member; a truncated node's `member_count == inspect(handle).edge_counts["members"]`. Covered by `tests/unit/mcp/operations/test_outline.py::TestOutlineMidExpansionTruncation` and the O.5 adversarial tests in `tests/conformance/test_linter_adversarial_outline.py`.

## 2026-06-21 — `expand(subclasses)` stubs are AST-built pointers (no Jedi re-derivation, no signature)

**Friction:** `resolve_subclasses` discarded the deterministic AST FQN `find_subclasses` already returns and re-derived a Jedi `Name` per subclass (`get_script` → `get_names` → `full_name` match), dropping any subclass whose `Name` it couldn't rebuild. Jedi's `full_name` is warm-inference-state-dependent (jedi #1796), so under artifact-cache thrash on Django (~2921 files) `expand(subclasses)` under-reported `find_subclasses`'s deterministic 1841 by 44–57% and drifted across result-cache rebuilds (1027 → 773 → 790) — re-importing the exact non-determinism #419 removed one layer down (#445).
**Decision:** Build each subclass adjacent as a `ClassSentinel` straight from the AST facts (`full_name` + file + `line`/`end_line`), never re-deriving a Jedi `Name` and never dropping. A subclass stub is a **pointer** (handle + location + kind) — the constructor signature is an `inspect` drill detail, not a one-hop `expand` field — symmetric with `imported_by`'s `ModuleSentinel`. The sentinel exposes `.type`/`.full_name` so it round-trips as a resolver INPUT at hop 2+ of `trace(follow=["subclasses"])`. Rejected: keep the Jedi `Name` when it resolves and fall back to a sentinel only on failure — that yields a deterministic *set* with non-deterministic *content* (signature flickers in/out across rebuilds), a subtler version of the same trust violation. Also rejected: re-add `signature` from the AST. A *faithful* signature needs an MRO walk — a subclass usually inherits its constructor (`inspect`/`members` already report the inherited `__init__` via Jedi, e.g. `DjangoPlugin(project_path: str)` though `DjangoPlugin` defines none), so the multi-file inference whose non-determinism #445 removed would come straight back; a *cheap* own-file-only signature is worse, present-but-wrong (`DjangoPlugin()`). Pointer-only is the honest stable contract, stated explicitly in the `stubs.py` docstring and the `python-explore` skill. `find_subclasses` now also carries `end_line` (`ClassDef.end_lineno`) and `get_end_line` honours an explicit `.end_line`, so the class span survives without Jedi.
**Anchor:** `resolve_subclasses` + `ClassSentinel` (`pyeye.mcp.operations.edges` / `pyeye._module_sentinel`); #445 (refs #419, #397, #333).
